## Key Points

- Traditional software testing methods fail with generative AI due to its stochastic and unpredictable nature, requiring enterprise-ready AI to adopt a new infrastructure layer called the AI Evaluation Stack
- The evaluation framework requires building two architectural layers: deterministic assertions (Layer 1) for structural validation and model-based assertions (Layer 2) for semantic quality assessment
- Deterministic assertions use traditional code and regex as a fail-fast first gate to validate structural integrity like JSON schemas, tool calls, and data formats before expensive semantic checks
- Model-based assertions (LLM-as-a-Judge) evaluate semantic quality using a superior reasoning model with strict rubrics and golden outputs for nuanced assessments that traditional code can't handle
- Evaluation requires dual pipelines: an offline pipeline for regression testing with golden datasets and an online pipeline for monitoring real-world behavior and detecting model drift
- The offline pipeline uses curated golden datasets of 200-500 test cases representing full operational envelopes, including edge cases and adversarial inputs for compliance requirements
- Teams must monitor five telemetry categories: explicit user signals, implicit behavioral signals, production deterministic asserts, asynchronous LLM-as-a-Judge evaluations, and feedback loop engineering
- High regeneration/retry rates, apology triggers, and refusal patterns serve as leading indicators of degraded AI capabilities or broken tool routing
- The continuous improvement workflow requires closed feedback loops that capture production failures, augment datasets, and run regression tests to avoid stagnant, stale evaluations
- Generative AI features are only "done" when rigorous automated evaluation pipelines are deployed and models consistently pass against both curated datasets and newly discovered production edge cases
