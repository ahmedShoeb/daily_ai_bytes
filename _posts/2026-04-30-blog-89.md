## Key Points

The author created an experimental Python harness (batching-harness) to test Anthopic's Batch API with individual agents, wrapping every turn in a single-entry batch
Anthropic's Batch API offers 50% discount on tokens but trades latency for cost savings - batches are asynchronous with up to 24-hour processing windows
With single-entry batches (parallel=1), turns take 90-120 seconds each, turning a five-turn agent loop into a 10-minute exercise - terrible for interactive use
Counterintuitively, the faster/cheaper Haiku model often takes longer in batches than Sonnet or Opus, possibly because fast processing leaves fewer idle windows for batch scheduling
This reverses conventional wisdom: if you're already accepting latency, you should use batch for expensive models (Opus) where the 50% discount has more absolute value
Batching only makes economic sense when: 1) Latency doesn't matter (overnight evals), 2) Running many agents in parallel, or 3) Amortizing across multiple harnesses
The most promising application is at fleet scale - pooling requests from many independent agents (CI runs, background subagents, team workflows) into shared batch submissions
Batch and prompt cache discounts stack, creating compounding effects especially valuable at fleet scale where shared prompt structure is common
The author is working on LunaRoute, a local proxy that could transparently route requests to batch or sync endpoints based on latency tolerance
Key insight: The effective unit for batching isn't a single user's turn, but a fleet's worth of turns pooled by an invisible layer
