## Key Points

- Infrastructure configuration can produce evaluation score differences exceeding typical leaderboard margins - up to 6 percentage points difference on Terminal-Bench 2.0 based on resource allocation
- Agentic coding evaluations (like SWE-bench and Terminal-Bench) are different from static benchmarks because the runtime environment becomes an integral component of the problem-solving process
- When container resource specs are enforced as both floor and hard ceiling (zero headroom), 5.8% of tasks fail due to infrastructure errors from transient memory spikes
- Success rates increase with resource headroom, with the biggest jump occurring between 3x resources and uncapped allocations (4 percentage point improvement)
- Below 3x resources, additional headroom mainly fixes infrastructure reliability problems; above 3x, resources actively help agents solve problems they couldn't solve before
- Resource configuration changes what evaluations measure: tight limits reward efficient strategies, while generous limits reward agents that exploit available resources
- The same pattern holds across different models and benchmarks, though magnitude varies (SWE-bench showed 1.54 percentage point improvement at 5x resources vs 1x)
- Infrastructure variables like time of day, cluster health, and API latency can also affect scores, blurring the line between 'model capability' and 'infrastructure behavior'
- The study recommends specifying both guaranteed allocation AND hard kill threshold parameters, with a calibrated buffer (like 3x) to reduce infrastructure noise without inflating scores
- Benchmark score differences below 3 percentage points deserve skepticism unless evaluation configurations are documented and matched
