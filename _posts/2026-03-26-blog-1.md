## Key Points

- Cursor is developing fast regex search indexes for AI agents to overcome the limitations of traditional grep/ripgrep which scan all files and become slow in large repositories
- The article traces the history of text indexing from 1973's grep through modern approaches like trigram decomposition, suffix arrays, and sparse n-grams
- Traditional trigram indexes (3-character sequences) work but have trade-offs - too few trigrams give large posting lists, too many give too many keys
- GitHub's Project Blackbird used probabilistic masks with bloom filters to add information about following characters while minimizing storage
- Sparse n-grams (used by ClickHouse and GitHub's new Code Search) extract variable-length substrings using deterministic weight functions, allowing fewer index lookups at query time
- Cursor builds indexes locally on user machines for lower latency, privacy, and to handle frequent updates when agents read their own writes
- The local index uses two files: one for posting lists, one for a sorted hash table, with memory-mapped lookup for efficient querying
- Performance improvements are most significant for large enterprise repositories where traditional grep operations can take 15+ seconds
- The fast regex search system is integrated with Cursor's Composer 2 model to create qualitative improvements in agentic coding workflows
