## Key Points

Anthropic's prompt caching system has three price tiers: 5-minute cache writes cost 1.25x base input, 1-hour cache writes cost 2x base, and cache reads cost 0.10x base input
There's a 62.5-minute rule for cache management: if you expect to need the cache again within 62.5 minutes, refresh it; if not, let it expire and rewrite later
Cache refresh operations serve two purposes: reading cached content (billed at read rate) and refreshing the cache TTL back to 5 minutes
The 62.5-minute rule is universal across all Claude models (Opus, Sonnet, Haiku) and cache sizes - the break-even point remains the same regardless of model or prefix size
Cache has minimum token requirements: Opus 4.5-4.7 needs at least 4,096 tokens, Sonnet 4.6 needs at least 1,024 tokens to enable caching
The cache lookback window only scans backward through 20 content blocks - if your agent adds more than 20 blocks between cache hits, the cache entry may fall outside the search window
Compacting context (summarizing conversations) has a cost-benefit ratio: you need roughly 10:1 compression for compaction to pay off in about 8 future turns
Opus 4.7 uses a new tokenizer that can increase token counts by up to 35% for the same text compared to earlier models
For a 500K token prefix on Opus, keeping cache warm for 30 minutes saves $1.63, but at 90 minutes it becomes $1.38 more expensive than letting it expire and rewriting
Cache economics depend heavily on usage patterns - if 30% of sessions ask only one question and leave, caching may not be worth it at all
