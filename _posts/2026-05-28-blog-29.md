## Key Points

The article explains how GPU tensor cores during LLM inference spend most of their time waiting due to memory bandwidth constraints, particularly during autoregressive decode phases where GPU HBM bandwidth is the bottleneck
This memory bottleneck has persisted and worsened across GPU generations as computational power has grown faster than memory bandwidth
The AI hardware market is organizing around different approaches to attack memory constraints: inside the chip (SRAM solutions), inside serving engines, inside cache hierarchies, or inside physical packaging
Companies like Groq use on-chip SRAM instead of HBM, Cerebras uses wafer-scale chips with massive internal memory bandwidth, while MatX and d-Matrix explore scratchpad memory and in-memory compute respectively
Software can mitigate hardware bottlenecks through scheduling techniques - a roofline heuristic shows that decode batch size scales roughly with the ratio of memory bandwidth to compute, with approximate batch size around 300 tokens for dense models or 6,000 for MoE architectures
KV cache memory consumption during inference grows linearly with context length and batch size, potentially reaching 100GB at 200k tokens, creating another memory challenge beyond weight reads
Companies like TensorMesh's LMCache address KV cache management by storing caches across GPU, CPU RAM, NVMe, and S3 storage
Packaging and interconnect challenges (CoWoS technology) become bottlenecks at scale, creating opportunities for companies solving mechanical engineering and materials science problems
Etched is betting on fixed-function silicon for transformers, while NVIDIA's Rubin platform now includes SRAM-based inference via the Groq acquisition
Algorithmic progress (speculative decoding, KV-cache compression, sparsity, low-bit models) changes the memory constraint but doesn't eliminate the underlying hardware bottleneck
