## Key Points

The author challenges the current industry trend of moving from Model Context Protocol (MCP) to CLIs, arguing this shift is driven by influencer hype cycles rather than practical engineering considerations.
There are token savings with CLI tools that exist in models' training datasets (like curl, git, grep), but these savings disappear when agents need to understand custom APIs or bespoke CLI tools.
MCP over stdio (local mode) may be unnecessary overhead, but MCP over streamable HTTP is transformative for organizational and enterprise adoption of AI agents.
Centralized MCP servers over HTTP enable: richer capabilities (access to databases, graph queries), better security (OAuth authentication, controlled secrets), telemetry/observability, and standardized content delivery.
MCP includes not just tools, but also prompts and resources - which function as server-delivered skills and documentation that stay automatically updated across an organization.
For enterprise use cases, MCP provides critical scaffolding for moving from individual 'vibe-coding' to structured, organizational agentic engineering with consistency, security, and maintainability.
The author emphasizes the need for engineering discipline in AI-generated software, drawing parallels to Amazon's AWS challenges with maintaining operationalized systems.
Influencers like Garry Tan and Andrew Ng provide advice based on individual usage, but their approaches may not scale to heterogeneous organizational environments.
Custom CLI tools face the same context problems as MCP tools but without the structured schema and organizational benefits.
MCP remains essential for organizations seeking to implement AI agent workflows at scale with proper security, telemetry, and maintainability considerations.
