## Key Points

OpenAI faces the challenge of delivering low-latency voice AI to over 900 million weekly active users globally
Key requirements include fast connection setup, stable media round-trip time with low jitter/packet loss, and global reach for natural voice conversations
OpenAI rearchitected its WebRTC stack to handle three scaling constraints: one-port-per-session limitations, stateful session ownership needs, and global routing requirements
The team developed a split relay plus transceiver architecture with a lightweight UDP forwarding layer and stateful WebRTC endpoint
This architecture uses the ICE username fragment (ufrag) as a protocol-native routing hook for first-packet routing before sessions exist
Global Relay distributes geographically to reduce first client-to-OpenAI hop latency by entering the network close to users
The relay-forward design preserves standard WebRTC behavior for clients while creating a smaller, more secure public UDP surface area
OpenAI chose a transceiver model over selective forwarding units (SFUs) as most sessions are 1:1 user-to-model interactions with high latency sensitivity
The implementation handles packet routing without protocol termination, keeping encryption and session state in the transceiver only
This infrastructure approach enables conversational AI that can begin processing while users are still talking, creating more natural voice interactions
